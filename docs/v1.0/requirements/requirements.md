# Textbook RAG Web - 需求文档

## 1. 项目概述

### 1.1 背景

当前项目目标是为 AI/ML/NLP 教材提供带深度溯源的问答能力。现有依赖以 Python 本地交互界面为主，难以支撑标准化 API、独立前端演进、以及更清晰的页面导航与状态反馈。

根据最新需求，项目调整为一个前后端分离的 Web 项目：

- 前端负责检索、问答、来源查看、文档浏览等用户交互
- 后端负责文档索引、混合检索、问答编排、溯源数据输出
- 两端通过明确的 HTTP API 通信

### 1.2 目标

- 将项目形态从单体式 Python UI 调整为标准前后端分离架构
- 保留并强化教材问答与深度来源追踪能力
- 为后续独立部署、测试、迭代和团队协作建立清晰边界

### 1.3 范围

本阶段覆盖需求定义，不锁定最终实现细节。明确纳入范围的能力：

- 教材文档问答
- 多路检索与结果融合
- 来源页码、章节、文本片段、坐标信息展示
- 面向浏览器的 Web 前端
- 面向前端与后续集成使用的后端 API

当前不纳入本阶段强制范围的能力：

- 用户登录与多租户
- 在线教材上传后台
- 生产级分布式部署
- ROS 2 集成细节

## 2. 目标用户

### 2.1 用户画像

- 学习 AI/ML/NLP 的学生
- 需要快速定位教材出处的助教或教师
- 需要验证答案可信度的项目评审者

### 2.2 用户场景

- 输入自然语言问题，获得基于教材的答案
- 查看答案对应的教材、章节、页码和命中区域
- 在同一页面左侧浏览 PDF，右侧进行问答交互
- 点击回答中的来源位置索引后，左侧 PDF 直接跳转到对应页和对应区域
- 从前端切换检索结果、来源页、引用内容，而不依赖 Python 本地页面
- 通过浏览器在桌面端完成演示与评估

## 3. 核心问题与痛点

- 现有项目素材与依赖已经具备 RAG 方向，但缺少清晰的 Web 产品边界
- 单体式 UI 不利于前后端职责分离、接口复用与自动化测试
- 来源追踪是核心卖点，需要独立的数据协议支持，而不是仅靠页面拼接
- 后续若继续扩展评估、演示或集成，缺少后端 API 会导致演进成本偏高

## 4. 功能需求

### 4.1 核心功能

1. 问答检索
   - 用户可输入问题并提交查询
   - 系统返回答案、置信信息或回答依据说明

2. 混合检索
   - 后端支持关键词检索、向量检索以及结构化过滤
   - 检索结果可统一排序并返回给前端

3. 深度来源追踪
   - 每条答案必须返回来源教材、章节、页码、片段摘要
   - 若存在版面坐标，需返回可用于前端高亮展示的区域信息

4. 来源浏览
   - 前端可展示来源列表
   - 用户可点击来源查看原始页与高亮区域

5. 双栏阅读问答界面
   - 前端提供左侧 PDF 浏览区域与右侧对话交互区域
   - 右侧回答中展示可点击的来源位置索引
   - 点击来源位置索引后，左侧 PDF 必须跳转到对应页码；若后端返回 bbox，则前端必须高亮对应区域；若当前阶段未提供 bbox，则必须至少完成页码定位

6. 前后端分离接口
   - 前端仅通过后端公开 API 获取数据
   - 前端不直接访问索引或本地文档处理逻辑

### 4.2 辅助功能

- 查询历史或最近问题列表
- 检索结果为空时的引导与纠错提示
- 可展示检索命中的多条候选来源
- 支持按书名、章节、内容类型进行过滤
- 支持当前 PDF 与当前问答上下文联动，减少用户在多个页面之间切换

### 4.3 未来扩展

- 用户认证与个性化收藏
- 教材管理后台
- 评测集管理与自动评分
- 与机器人或外部系统的 API 集成

## 5. 非功能需求

### 5.1 可用性

- 首页必须让用户快速理解“能问什么、怎么查看来源、下一步去哪”
- 导航结构要稳定，关键入口位置一致
- 查询、加载、失败、空结果都需要明确反馈
- PDF 阅读区与对话区应同时可见，避免用户在阅读和问答之间反复切页
- 来源索引的点击反馈必须明确，用户能立即感知 PDF 已跳转到新位置

### 5.2 性能

- 常规查询应在可接受的交互等待时间内返回首屏结果，具体阈值在 PRD 阶段确定
- 前端需支持渐进展示答案与来源，不阻塞整个页面
- 后端接口应支持分页或限制返回量，避免一次性回传过大数据

### 5.3 可维护性

- 前端与后端代码库边界清晰
- 接口契约明确，可独立测试
- 数据模型、来源模型、检索模型要可扩展

### 5.4 可靠性

- 后端在模型不可用、检索为空、文档缺失时需返回可处理错误
- 前端对超时、失败、空响应要有稳定降级

### 5.5 可测试性

- 后端核心 API 需要可自动化测试
- 前端关键交互需具备基础页面或组件级验证
- 需求与接口应能支撑后续评测问题集

## 6. 约束条件

### 6.1 技术约束

- 仓库当前以 Python 为主，已有 FastAPI 依赖，可优先作为后端基础
- 前端框架尚未确定，在架构阶段确定
- 必须保留现有教材数据、MinerU 结果、topic index 与检索相关资产

### 6.2 业务约束

- 项目仍需服务课程作业场景，强调演示效果与可解释性
- 深度来源追踪是差异化能力，不能在改造中丢失

### 6.3 时间约束

- 现阶段优先完成可演示 MVP
- 先完成前后端分层与核心问答链路，再考虑扩展功能

## 7. 验收标准

1. 项目存在独立的前端与后端目录或等价的清晰工程边界
2. 浏览器前端能够调用后端 API 完成一次教材问答
3. 返回结果中包含可展示的来源元数据
4. 前端页面支持左侧 PDF 浏览和右侧问答交互同时存在
5. 用户点击回答中的来源位置索引后，左侧 PDF 会跳转到对应页或对应区域
6. 用户可以从前端查看至少一个来源页或来源详情
7. 当查询失败或无结果时，前端能够给出明确反馈
8. 后续文档阶段能够在本需求基础上继续产出 PRD 与架构文档

## 8. 假设与待确认项

- 假设后端采用 FastAPI，因仓库已有相关依赖
- 假设前端将使用独立 Web 前端而非 Streamlit 作为主界面
- 待确认是否保留 Streamlit 作为临时调试页
- 待确认前端技术栈是 React/Vite 还是继续采用 Python Web UI 方案作为过渡
- 待确认 PDF 跳转是否采用页码 + bbox 高亮，还是仅先做页码定位的 MVP

## 9. 教材依据

- Ref: Steve Krug, *Don’t Make Me Think*, navigation and persistent navigation. 需求强调稳定导航、明确首页入口与低认知负担。
- Ref: Don Norman, *The Design of Everyday Things*, discoverability, feedback, conceptual models. 需求强调可发现操作、明确系统反馈、让前后端职责对用户心智一致。
- Ref: Winters et al., *Software Engineering at Google*, time/scale/trade-offs and documentation. 需求强调前后端边界清晰、接口可演进、文档先行以支撑后续长期维护。
